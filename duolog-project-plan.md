# Duolog — 结构化深度对话工具

> 帮两个人聊到他们迟早会撞上、但从来不会主动聊的话题。

## 项目命名

**Duolog** = Duo（二人）+ Dialogue（对话）

定位不是"婚姻兼容性测试"，而是"情侣/伴侣深度对话的脚手架"。产品语言避开"评分""测试""合不合适"，用"发现""探索""对话"替代。

备选名：Resonance（共振）、PairLens、深聊 DeepTalk

---

## 核心价值主张

大多数情侣在进入长期关系前，从未就关键人生议题做过结构化对话。不是不想聊，是不知道该聊什么、怎么开口。

Duolog 通过具体场景题，帮两个人在 10 分钟内发现彼此在 7 个关键维度上的对齐程度和分歧点，并提供"接下来怎么聊"的引导。

## 技术选型

| 层级 | 选择 | 理由 |
|------|------|------|
| 框架 | React 18 + TypeScript | 组件化开发，类型安全，生态成熟 |
| 构建 | Vite | 快速冷启动，HMR 体验好，配置极简 |
| 样式 | Tailwind CSS | 原子化 CSS，快速迭代，bundle 小 |
| 图表 | Recharts | 轻量，React 原生，雷达图支持好 |
| 路由 | React Router v6 | 单页应用标准方案 |
| 国际化 | i18next | 中英双语，后续扩展多语言 |
| 部署 | Vercel / Cloudflare Pages | 零配置部署，免费额度足够 |
| 数据 | 纯前端，localStorage | v1 不需要后端，数据留在用户设备 |

**为什么不用纯 HTML/JS：** 纯 HTML 可以做 v0.1 原型，但一旦涉及多页面路由、状态管理（两人答题流程）、组件复用（题目卡片、雷达图、结果面板），React 的工程效率远高于手写 DOM 操作。开源项目要让别人能贡献代码，组件化架构的可维护性也好得多。

## 分阶段路线图

### Phase 0 — MVP（1-2 周）

目标：可部署的最小可用版本，验证核心体验。

- 把现有 JSX 迁移到 Vite + React + TS 项目
- 7 维度 × 3 题 = 21 题的完整流程
- 双人答题 → 雷达图对比 → 维度分析
- localStorage 保存结果（刷新不丢失）
- 部署到 Vercel，拿到可分享链接
- 中文版优先

交付物：可访问的线上 URL + GitHub 仓库 public

### Phase 1 — 体验打磨（2-4 周）

目标：从"能用"到"好用"。

- 答题动画和过渡效果（页面切换、选项反馈）
- 结果页增加"对话引导"模块：每个分歧维度给出 2-3 个具体对话 prompt
- 结果可导出为图片（用于社交分享）
- 移动端体验优化（触控友好、字体大小）
- 英文版上线
- SEO 基础（meta tags、og image）

### Phase 2 — 内容扩展（1-2 月）

目标：提升评估信度，增加可玩性。

- 每个维度扩展到 5-7 题（总题量 35-49 题），支持随机抽取子集
- 加权评分系统（核心分歧项如"是否要孩子"权重更高）
- 增加"已婚版"题库（关注共同生活中的具体场景）
- 增加"朋友/合伙人版"（商业合伙兼容性评估）
- 社区贡献题目流程：GitHub Issue 模板 → 审核 → 合并

### Phase 3 — 产品化（2-3 月）

目标：从开源工具到有传播力的产品。

- 历史记录：多次测试结果对比（看关系变化趋势）
- 匿名统计仪表盘：各维度的全局分布（"你们的金钱观对齐程度超过 72% 的用户"）
- 可选的 LLM 个性化解读（接 Claude API，根据具体答题模式生成定制分析）
- PWA 支持（添加到主屏幕）
- 小红书/微信生态的分享适配

### Phase 4 — 生态（远期）

- 题库开放 API，允许第三方嵌入
- 心理咨询师合作模式（咨询师可创建自定义题库）
- 配套"深度对话卡牌"实体产品（结合线下场景）
- 数据研究（匿名化后的维度相关性分析、文化差异研究）

## 评分方法论

### 当前方案（v1）

每道题 4 个选项，每个选项映射到 1-10 的光谱值。光谱代表的是倾向方向，不是好坏。

例如"风险态度"维度：
- 低分端 → 安全导向，偏好确定性
- 高分端 → 机会导向，能接受不确定性

每个维度取该维度下所有题目的平均分。两人的兼容性 = 各维度分差的加权平均。

分差阈值：
- ≤ 1.5 → 高度对齐（绿色）
- 1.5 - 3.0 → 存在差异（黄色）
- \> 3.0 → 显著分歧（红色）

### 待优化（v2+）

- 引入维度权重（核心不可妥协项 × 1.5，可协商项 × 1.0）
- 内部一致性检验（同一维度的题目之间 Cronbach's α）
- 社会期望偏差校正（检测是否在选"政治正确"的答案）
- 测试-重测信度验证（同一对用户间隔 2 周再测，结果稳定性）

## 题目设计原则

1. **场景化而非态度化**：问"你会怎么做"而非"你觉得重不重要"。行为预测优于态度声明。

2. **无标准答案**：每个选项都是合理的生活选择，只是方向不同。避免明显的"正确答案"诱导。

3. **高区分度**：四个选项之间应该有足够的行为差异，不能让 3 个选项实质相同。

4. **文化适应性**：题目场景应在目标文化中普遍可理解。中英版本不是直译，而是场景本地化。

5. **生活化语言**：不用心理学术语，不用抽象概念。读完题目应该像在跟朋友聊天。

## 学术参考

- Gottman, J. M. — 婚姻稳定性预测模型（修复尝试成功率、四骑士理论）
- Prepare/Enrich — 婚前评估工具的维度设计
- Kahneman & Tversky — 前景理论（参照系对满意度的影响）
- Killingsworth (2021) — 收入与幸福感的关系修正
- Schwartz, B. — 选择悖论（决策框架设计）
- Naval Ravikant — 重大决策的不可逆性思考框架

## 开源策略

- 协议：MIT（最大化传播和复用）
- 语言：代码和 commit message 英文，文档中英双语
- 贡献重点：最欢迎的贡献是新题目和翻译，其次是 UI/UX 优化
- Issue 标签体系：`question-proposal`、`i18n`、`ux`、`scoring`、`bug`
- README 风格：开头一句话说清楚是什么 → 截图 → 快速开始 → 贡献指南链接
